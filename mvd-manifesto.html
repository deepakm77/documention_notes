The "Minimum Viable Documentation" Manifesto

I Slashed My Team's Onboarding Time by 70% by Writing Less Documentation — Not More.

The new engineer joined on a Monday. By Wednesday, she had read forty-seven internal documents. By Friday, she still didn't know how to run the project locally.

Every doc contradicted the one before it. One README said to use docker-compose up. Another said that command was deprecated. A third mentioned a setup script that no longer existed. A Confluence page titled "Getting Started — UPDATED Jan 2024" had a comment from March that said "this is wrong, see the new guide" with a link that returned a 404.

She came to me at the end of her first week and said, almost apologetically: "I think the docs might be... broken?"

They weren't broken. They were worse than broken. They were confidently, authoritatively, professionally wrong. And there were hundreds of them.

I didn't assign someone to fix the docs. I declared war on them.

The Documentation Debt Nobody Talks About

Every engineering team talks about technical debt. Dead code, legacy systems, tangled dependencies — we have entire frameworks for managing these things. We have linters, static analysis, code reviews.

Nobody talks about documentation debt. And it's just as real, just as expensive, and dramatically harder to detect — because outdated documentation doesn't throw an error. It just misleads you quietly, confidently, in complete sentences.

Here's the paradox that kills teams: the more documentation you write, the more dangerous your docs become over time. Every new doc is a liability the moment the system it describes changes. And systems always change.

Outdated documentation isn't neutral. It's actively worse than no documentation. A blank page tells you nothing. A wrong page sends you in the wrong direction for three hours.

We had a FinTech payments module with eleven separate documents explaining how the retry logic worked. Written at different times, by different engineers, for different audiences. No single document was complete. No two documents agreed. The system itself had been refactored twice since the oldest doc was written.

Every new engineer who touched that module spent an average of two days reading docs before they could do anything productive. And at the end of those two days, they understood the docs — not the system.

That's when I introduced what I now call the Minimum Viable Documentation (MVD) Manifesto.

The Three Laws of MVD

Before I tell you what we built, let me tell you the philosophy behind it. The MVD Manifesto runs on three laws that sound obvious but are almost universally violated in practice.

Law 1: Documentation that isn't maintained is a bug, not a feature. Treat stale docs with the same urgency you'd treat broken tests. They're not "better than nothing." They're worse.

Law 2: The only documentation that earns its existence is documentation that answers a question someone is actually asking. Not a question someone might ask. Not a question that seemed important when the doc was written. A real question, asked today, by a real person on your team.

Law 3: If the code can say it, the code should say it. Docs that describe what the code does are always racing against the code and losing. Code that describes why it does something is irreplaceable.

These three laws led us to three concrete interventions.

Intervention 1: The Golden README

We burned everything down to a single document per project. One file. One source of truth. We called it the Golden README.

The Golden README answers exactly three questions. Not four. Not two. Three.

⭐ The Golden README — Three Questions Only

1
"What is this?" — One paragraph. What does this service do, why does it exist, and what does it not do. If you can't explain it in a paragraph, the service might be the problem.
2
"How do I run it locally?" — The exact commands, in order, copy-paste ready. No assumptions. No "you'll need to set up X first" without telling them how. If it takes more than five steps, the setup process might be the problem.
3
"Where do I look when things break?" — Links to the monitoring dashboard, the on-call runbook, the most common failure modes. Not an explanation of every possible failure. Just: where do you start when it's 2 AM and something is on fire.

Everything else — architecture decisions, historical context, edge case explanations — lives either in the code itself, in ADRs (Architecture Decision Records) with explicit dates, or nowhere at all.

Here's what our Golden README template looks like:

# payments-retry-service

## What is this?
Handles automatic retry logic for failed payment transactions.
Sits between the payment gateway and our internal ledger.
Does NOT handle refunds or disputes — see `refund-service` for that.

## Run it locally
cp .env.example .env
docker-compose up -d postgres redis
pip install -r requirements.txt
python main.py
# Service runs on http://localhost:8001

## When things break
- Logs: https://grafana.internal/payments-retry
- Runbook: https://notion.so/payments-retry-runbook
- On-call: #payments-oncall on Slack
- Most common issue: Redis connection timeout → restart Redis container

That's it. Thirty lines. Everything a new engineer needs to be dangerous in the first hour. Every word earns its place or it doesn't survive the next review.

The rule we enforced: if a question isn't answered by the Golden README and it's been asked twice, it gets added. If a section hasn't been referenced in six months, it gets deleted. The README is a living document with a strict diet.

Intervention 2: Code as Documentation

The most durable documentation in any codebase isn't a README. It's a well-named function.

We introduced a PR review policy: no merge if the code requires a comment to explain what it's doing. That's a naming problem, not a documentation problem. Rename the function until the code explains itself.

The only comments we accepted were comments that explained why — the business reason, the edge case that burned someone in production, the constraint imposed by an upstream system that makes no sense but is real.

The difference looks like this:

# BAD: comment explains what (the code should do this itself)
# multiply amount by 100 to convert to paise
def process(amount):
    return amount * 100

# GOOD: function name explains what, comment explains why
def convert_rupees_to_paise(amount):
    # RazorPay API requires amounts in smallest currency unit.
    # Sending rupees directly causes silent truncation on their end.
    # See: https://razorpay.com/docs/payments/amount-format
    return amount * 100

The second version doesn't need a doc. It is a doc. Six months from now, when someone wonders why this multiplication is here, the answer is in the code — timestamped, version-controlled, and guaranteed to be in the right place.

We started rejecting PRs for documentation violations the same way we rejected them for failing tests. It felt harsh for the first two weeks. By the fourth week, the team's code was noticeably cleaner and the PR review conversations had shifted from "what does this do?" to "why did we make this tradeoff?"

That's the conversation you want your team having.

Intervention 3: The Deletion Dash

This is the one people look at me funny about. Every month, we ran a one-day sprint with a single goal: delete as much documentation as possible.

We called it the Deletion Dash. The rules were simple:

🗑️ Deletion Dash — Rules

  • 📋  Any doc not updated in 6 months is a deletion candidate. Default is delete, not keep.
  • 🔗  Any doc that duplicates information available in code or in another doc gets deleted.
  • ❓  If you're unsure whether a doc is still accurate, you don't update it. You delete it and let someone recreate it from scratch if it's truly needed.
  • 🏆  The engineer who deletes the most words wins. (Yes, we tracked it. Yes, it became competitive.)

The first Deletion Dash was genuinely uncomfortable. Engineers kept saying things like "but what if someone needs this?" My answer every time: "If they need it, they'll create it. If they don't create it, they didn't need it."

In the first dash, we deleted over 60 documents and roughly 40,000 words of internal documentation. The Confluence space went from 200+ pages to 31. The wiki went from a maze to a map.

Nobody came back the next day asking where their doc had gone. Not a single person.

The most revealing thing about the Deletion Dash isn't what gets deleted. It's that nobody notices. That silence is data. It tells you everything about which documentation was actually being read.

Three weeks to four days. That's not a rounding error. That's an engineer going from theoretical to productive in less than a week instead of less than a month.

But the metric I'm proudest of isn't the onboarding time. It's this: engineers started voluntarily updating documentation. That had never happened before. When the docs are small enough to be maintainable and accurate enough to be trustworthy, people treat them like something worth maintaining. When docs are a sprawling graveyard of outdated information, nobody wants to touch them — and for good reason.

We also saw a drop in "documentation questions" in Slack — the messages that start with "hey, I read the doc but I'm confused about..." Those dropped by around 80% in the quarter after we implemented MVD. Not because people stopped having questions, but because the docs now actually answered the questions people were asking.

The Hardest Pushback I Got

"But what about knowledge transfer? What happens when someone leaves?"

Fair question. Here's my answer: undocumented knowledge that lives in a person's head is dangerous. But that knowledge doesn't transfer to a doc that nobody reads. It transfers to a doc that someone actually opens, or to clean, self-explanatory code, or to a brief Slack thread that gets pinned in the right channel.

The engineers who leave your team aren't usually leaving behind knowledge gaps because they didn't write enough. They're leaving behind gaps because what they knew was tribal, context-dependent, and impossible to write down in a way that survives them. The answer to that isn't more documentation. It's better code, better architecture, and the discipline to keep the Golden README accurate.

One more pushback I hear: "This doesn't scale." Actually, bloated documentation is what doesn't scale. Every doc you add is a doc someone has to read, maintain, and eventually delete. The bigger your team gets, the more expensive your documentation sprawl becomes. MVD scales precisely because it demands that documentation shrink as the team grows, not expand.

The MVD Manifesto — In One Place

The Rules

  • One Golden README per project. Answers exactly three questions. Nothing else.
  • Code explains what. Comments explain why. Reject PRs that need comments to explain logic that should be in the function name.
  • Delete first, add second. Before writing any new documentation, delete something first.
  • Run a monthly Deletion Dash. Default is delete. The burden of proof is on the document to justify its existence.
  • Docs have expiry dates. Every document gets a "last verified" date. Anything over six months old is a deletion candidate by default.
  • If it's been asked twice, it goes in the README. If a README section hasn't been referenced in six months, it comes out.

What This Is Really About

Documentation bloat is a symptom of a deeper problem: teams that confuse writing with communicating. Writing a document feels productive. It feels responsible. It feels like you're doing the right thing for the next engineer who has to understand your system.

But a document nobody reads is not communication. It's performance. It's the engineering equivalent of a meeting that should have been an email — except somehow worse, because it actively misleads the people who do try to engage with it.

The Minimum Viable Documentation Manifesto is, at its core, about respecting your reader's time more than your own desire to feel like you've documented something. It's harder to write thirty accurate lines than three hundred vague ones. Constraint forces clarity. Clarity builds trust. Trust means your team actually reads what you write.

And when your team actually reads what you write, something remarkable happens: documentation stops being a chore and starts being a tool.

Write less.
Delete more.
Make every word earn its place.

The best documentation you'll ever write
might be the documentation you choose not to write.
